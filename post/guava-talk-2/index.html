Guava杂谈 - EventBus

Tuesday, June 23, 2020

EventBus俗称“事件总线”。它的作用方式就如上图所示，能够将数据推送到所有需要它的人订阅者手上。它的行为总共分为4步：

  1. Subscriber向EventBus订阅某种Event
  2. Publisher向EventBus发送某种Event
  3. EventBus将Event送到订阅了它的Subscriber手上
  4. Subscriber执行相应操作

而在这简单的行为模式中，EventBus又干了多少我们不知道的事呢？

从观察者模式开始

基于一个对象的动作而引起与之关联的所有对象都做出各自的反应，这个时候我们就会使用到观察者模式。

如果我们将上述EventBus中的事件类型删掉，假定所有的事件就是同一类型，我们就能够很轻松的用观察者模式实现这一功能。

// CustomEventBus.java
public class CustomEventBus {
    private List<Subscriber> subscriberList;

    public void subscribe(Subscriber subscriber) {
        if(Objects.isNull(subscriberList)) {
            subscriberList = Lists.newArrayList();
        }
        subscriberList.add(subscriber);
    }

    public void post() {
        if(Objects.isNull(subscriberList)) return;
        subscriberList.forEach(Subscriber::update);
    }

}
// Subscriber.java
public interface Subscriber {
    public void update();
}
// SubscriberA.java
public class SubscriberA implements Subscriber {
    @Override
    public void update() {
        System.out.println("This is subscriber a");
    }
}
// SubscriberB.java
public class SubscriberB implements Subscriber {
    @Override
    public void update() {
        System.out.println("This is subscriber b");
    }
}
// CustomEventBusTest.java
public class CustomEventBusTest {
    @Test
    public void postTest() {
        CustomEventBus eventBus = new CustomEventBus();
        Subscriber a = new SubscriberA();
        Subscriber b = new SubscriberB();
        eventBus.subscribe(a);
        eventBus.subscribe(b);
        eventBus.post();
    }
}
// 输出结果
This is subscriber a
This is subscriber b

以上的实现十分的简单，但从功能上而言没有事件类型的支持而显得没有那么实用，因此，接下来我们尝试让上述代码支持不同的事件类型。

泛型和反射

如果单纯从整体的功能而言，事件类型的支持有两种方式：

  1. EventBus无差别将所有事件传递给所有Subscriber，Subscriber自身进行类型筛选
  2. EventBus只传递Subscriber订阅的事件类型

但从实际职责划分来看，筛选Event这件事情不应该在“处理事件”逻辑里，而应该在“分发事件”里。并且，如果Subscriber接收所有类型的事件，毫无疑问会导致整体的处理能力下降。因此，只能选择方法2。

那么，如何让Subscriber在订阅的时候告知EventBus它所需要的类型并且在Subscriber的update()方法中也要传递不同类型的Event呢？答案就是使用泛型。

// Subscriber.java
public interface Subscriber<T> {
    public void update(T event);
}

而在接口使用泛型的前提下，EventBus需要做到三点：

  1. 通过Subscriber子类的实例，获取它所使用的具体类型
  2. 关联Event类型和Subscriber实例
  3. post()方法使用的参数必须能够传递给Subscriber.update()方法
public class CustomEventBus {
    // 通过Map管理类和相应Subscriber
    private Map<Class<?>, Set<Subscriber<?>>> subscriberMap;

    public void subscribe(Subscriber<?> subscriber) {
        if(Objects.isNull(subscriber)) return;
        if(Objects.isNull(subscriberMap)) {
            subscriberMap = Maps.newHashMap();
        }
        // 获取接口所使用的泛型
        ParameterizedType parameterizedType = (ParameterizedType)subscriber.getClass().getGenericInterfaces()[0];
        Class<?> eventClazz = (Class<?>)parameterizedType.getActualTypeArguments()[0];

        subscriberMap.putIfAbsent(eventClazz, Sets.newHashSet());
        subscriberMap.get(eventClazz).add(subscriber);
    }

    public void post(Object event) {
        if(Objects.isNull(subscriberMap)
                || Objects.isNull(event)
                || Objects.isNull(subscriberMap.get(event.getClass())))
            return;
        Set<Subscriber<?>> subscriberSet = subscriberMap.get(event.getClass());
        for(Subscriber<?> subscriber : subscriberSet) {
            try {
                // 使用反射调用update方法
                Method updateMethod = subscriber.getClass().getMethod("update", event.getClass());
                updateMethod.invoke(subscriber, event);
            } catch (Exception e) {
                e.printStackTrace();
            }
        }
    }
}

上述代码相较于前一版修改了以下几点：

  1. 存储Subscriber的容器由List改为Map
  2. 存储同一类型事件的Subscriber使用了Set，保证一个Subscriber实例仅调用一次upate()
  3. 使用反射获取子类具体所选择的泛型，建立事件和Subscriber的关联
  4. 使用反射调用update()方法，使得传入post()的事件能够正确向下转型

Java的反射的强大之处就体现了出来，它能够是你在很多时候在实现时一定程度上打破类型带来的限制，更加灵活的去进行抽象和统一操作。

接口和注解

接口是什么？它代表的是一类操作的统一抽象。并且，接口的定义本身也是一种元信息的表示，它能够表示的是实现（子类）是抽象（接口）的一种，这也是多态的来源。但在java中，元信息的表示不仅仅只有继承接口这一种，还有一种方式是通过注解去标识某个类、方法、变量的元信息。

从上述例子上来看，接口作为抽象有时候会显得有些“臃肿”，定义一个接口仅仅就是为了一个方法的抽象，在Subscriber很多的时候需要很多的实现类。与此相类似的例子还有SpringMVC中的Controller接口，固定类型的参数和返回值、一个类映射一个url，而取代这一方式的就是如今的基于注解的Controller映射，不仅能够使url的映射变为方法级，连参数和返回值相较之前也变得更为灵活。因此，EventBus也同样能够通过注解的方式去解决这一问题。

// Listen.java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Listener {
}

首先，定义一个用于修饰方法的注解，如果方法使用了这个注解，那么将会在该方法的参数类型的时间传递时被调用。

public class Subscriber {

    private Object targetObject;
    private Method updateMethod;

    public void setTargetObject(Object o) {
        this.targetObject = o;
    }

    public void setUpdateMethod(Method method) {
        this.updateMethod = method;
    }

    public void update(Object event) {
        try{
            updateMethod.invoke(targetObject, event);
        } catch (IllegalAccessException | InvocationTargetException e) {
            e.printStackTrace();
        }
    }
}

其次，是方法抽取为单独的类，由于Java的最小单位是类，方法无法脱离类而存在，而Method的调用除了方法对象本身之外，还需要目标对象。通过抽取这个类，我们就能够在每识别到一个符合要求的方法之后，将之抽取为单独的Subscriber对象，最后保存在Map当中。

// CustomEventBus.java
public class CustomEventBus {
    // 通过Map管理类和相应Subscriber
    private Map<Class<?>, Set<Subscriber>> subscriberMap;

    public void subscribe(Object subscriber) {
        if(Objects.isNull(subscriber)) return;
        if(Objects.isNull(subscriberMap)) {
            subscriberMap = Maps.newHashMap();
        }
        Method[] methodArray = subscriber.getClass().getMethods();
        for(Method method: methodArray) {
            // 判断方法是否被@Listener修饰
            if(method.isAnnotationPresent(Listener.class)) {
                Subscriber sub = new Subscriber();
                sub.setTargetObject(subscriber);
                sub.setUpdateMethod(method);
                // 通过方法参数获取事件类型
                Class<?> eventClazz = method.getParameterTypes()[0];
                subscriberMap.putIfAbsent(eventClazz, Sets.newHashSet());
                subscriberMap.get(eventClazz).add(sub);
            }
        }
    }

    public void post(Object event) {
        if(Objects.isNull(subscriberMap)
                || Objects.isNull(event)
                || Objects.isNull(subscriberMap.get(event.getClass())))
            return;
        Set<Subscriber> subscriberSet = subscriberMap.get(event.getClass());
        for(Subscriber subscriber : subscriberSet) {
            // 使用反射调用update方法
            subscriber.update(event);
        }
    }
}

上述的CustomEventBus和前一版的实现不同在于：

  1. 不再使用泛型表示事件类型，使用@Listener注解了的方法，其参数作为其监听的对象类型
  2. 调用update()方法不再使用反射，所有反射操作在构建Subscriber对象时完成

注解的使用是Java的又一大杀器，它能够让你突破面向对象编程的限制去进行元信息的添加，使用得当可以大幅度提高框架的灵活性。

总结

今天讨论的大部分内容其实是反射和注解在Java中扮演的角色。注解和反射作为Java区别于其他语言的两大特性，能够很好的帮助框架设计者做出更灵活的框架，但它们的使用远比上面所写的要复杂，并且反射也是一种比较“危险”的操作，在实际使用时需要慎重考虑并处理好相应的异常，否则很容易发生灾难。

上述的代码仅仅作为理解EventBus的一部分原理而存在，实际的实现更为复杂和精密，后续还会讨论。
