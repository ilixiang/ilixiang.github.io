Guava杂谈 - EventBus

2020年6月23日
| Java
| 阅读

EventBus俗称“事件总线”。它的作用方式就如上图所示，能够将数据推送到所有需要它的人订阅者手上。它的行为总共分为4步：

  1. Subscriber向EventBus订阅某种Event
  2. Publisher向EventBus发送某种Event
  3. EventBus将Event送到订阅了它的Subscriber手上
  4. Subscriber执行相应操作

而在这简单的行为模式中，EventBus又干了多少我们不知道的事呢？

从观察者模式开始

基于一个对象的动作而引起与之关联的所有对象都做出各自的反应，这个时候我们就会使用到观察者模式。

如果我们将上述EventBus中的事件类型删掉，假定所有的事件就是同一类型，我们就能够很轻松的用观察者模式实现这一功能。

// CustomEventBus.java
public class CustomEventBus {
    private List<Subscriber> subscriberList;

    public void subscribe(Subscriber subscriber) {
        if(Objects.isNull(subscriberList)) {
            subscriberList = Lists.newArrayList();
        }
        subscriberList.add(subscriber);
    }

    public void post() {
        if(Objects.isNull(subscriberList)) return;
        subscriberList.forEach(Subscriber::update);
    }

}

public interface Subscriber {

}
// SubscriberA.java
public class SubscriberA implements Subscriber {
    @Override
    public void update() {
        System.out.println("This is subscriber a");
    }
}
// SubscriberB.java
public class SubscriberB implements Subscriber {
    @Override
    public void update() {

        eventBus.subscribe(b);
        eventBus.post();
    }

This is subscriber b

以上的实现十分的简单，但从功能上而言没有事件类型的支持而显得没有那么实用，因此，接下来我们尝试让上述代码支持不同的事件类型。

泛型和反射

如果单纯从整体的功能而言，事件类型的支持有两种方式：

  1. EventBus无差别将所有事件传递给所有Subscriber，Subscriber自身进行类型筛选
  2. EventBus只传递Subscriber订阅的事件类型

但从实际职责划分来看，筛选Event这件事情不应该在“处理事件”逻辑里，而应该在“分发事件”里。并且，如果Subscriber接收所有类型的事件，毫无疑问会导致整体的处理能力下降。因此，只能选择方法2。

那么，如何让Subscriber在订阅的时候告知EventBus它所需要的类型并且在Subscriber的update()方法中也要传递不同类型的Event呢？答案就是使用泛型。

  1. 通过Subscriber子类的实例，获取它所使用的具体类型
  2. 关联Event类型和Subscriber实例
  3. post()方法使用的参数必须能够传递给Subscriber.update()方法

public class CustomEventBus {
    // 通过Map管理类和相应Subscriber
    private Map<Class<?>, Set<Subscriber<?>>> subscriberMap;

    public void subscribe(Subscriber<?> subscriber) {
        if(Objects.isNull(subscriber)) return;
        if(Objects.isNull(subscriberMap)) {
            subscriberMap = Maps.newHashMap();
        }
        // 获取接口所使用的泛型
        ParameterizedType parameterizedType = (ParameterizedType)subscriber.getClass().getGenericInterfaces()[0];
        Class<?> eventClazz = (Class<?>)parameterizedType.getActualTypeArguments()[0];

        subscriberMap.putIfAbsent(eventClazz, Sets.newHashSet());
        subscriberMap.get(eventClazz).add(subscriber);
    }

    public void post(Object event) {
        if(Objects.isNull(subscriberMap)
                || Objects.isNull(event)
                || Objects.isNull(subscriberMap.get(event.getClass())))
            return;
        Set<Subscriber<?>> subscriberSet = subscriberMap.get(event.getClass());
        for(Subscriber<?> subscriber : subscriberSet) {
            try {
                // 使用反射调用update方法
                Method updateMethod = subscriber.getClass().getMethod("update", event.getClass());
                updateMethod.invoke(subscriber, event);
            } catch (Exception e) {
                e.printStackTrace();
            }
        }
    }
}

上述代码相较于前一版修改了以下几点：

  1. 存储Subscriber的容器由List改为Map
  2. 存储同一类型事件的Subscriber使用了Set，保证一个Subscriber实例仅调用一次upate()
  3. 使用反射获取子类具体所选择的泛型，建立事件和Subscriber的关联
  4. 使用反射调用update()方法，使得传入post()的事件能够正确向下转型

Java的反射的强大之处就体现了出来，它能够是你在很多时候在实现时一定程度上打破类型带来的限制，更加灵活的去进行抽象和统一操作。

接口和注解

接口是什么？它代表的是一类操作的统一抽象。并且，接口的定义本身也是一种元信息的表示，它能够表示的是实现（子类）是抽象（接口）的一种，这也是多态的来源。但在java中，元信息的表示不仅仅只有继承接口这一种，还有一种方式是通过注解去标识某个类、方法、变量的元信息。

从上述例子上来看，接口作为抽象有时候会显得有些“臃肿”，定义一个接口仅仅就是为了一个方法的抽象，在Subscriber很多的时候需要很多的实现类。与此相类似的例子还有SpringMVC中的Controller接口，固定类型的参数和返回值、一个类映射一个url，而取代这一方式的就是如今的基于注解的Controller映射，不仅能够使url的映射变为方法级，连参数和返回值相较之前也变得更为灵活。因此，EventBus也同样能够通过注解的方式去解决这一问题。

public class Subscriber {

    private Object targetObject;
    private Method updateMethod;

    public void setTargetObject(Object o) {
        this.targetObject = o;

// CustomEventBus.java
public class CustomEventBus {
    // 通过Map管理类和相应Subscriber
    private Map<Class<?>, Set<Subscriber>> subscriberMap;

    public void subscribe(Object subscriber) {
        if(Objects.isNull(subscriber)) return;
        if(Objects.isNull(subscriberMap)) {
            subscriberMap = Maps.newHashMap();
        }
        Method[] methodArray = subscriber.getClass().getMethods();
        for(Method method: methodArray) {
            // 判断方法是否被@Listener修饰
            if(method.isAnnotationPresent(Listener.class)) {
                Subscriber sub = new Subscriber();
                sub.setTargetObject(subscriber);
                sub.setUpdateMethod(method);
                // 通过方法参数获取事件类型
                Class<?> eventClazz = method.getParameterTypes()[0];
                subscriberMap.putIfAbsent(eventClazz, Sets.newHashSet());
                subscriberMap.get(eventClazz).add(sub);
            }
        }
    }

    public void post(Object event) {
        if(Objects.isNull(subscriberMap)
                || Objects.isNull(event)
                || Objects.isNull(subscriberMap.get(event.getClass())))
            return;
        Set<Subscriber> subscriberSet = subscriberMap.get(event.getClass());
        for(Subscriber subscriber : subscriberSet) {
            // 使用反射调用update方法
            subscriber.update(event);
        }
    }
}

上述的CustomEventBus和前一版的实现不同在于：

  1. 不再使用泛型表示事件类型，使用@Listener注解了的方法，其参数作为其监听的对象类型
  2. 调用update()方法不再使用反射，所有反射操作在构建Subscriber对象时完成

注解的使用是Java的又一大杀器，它能够让你突破面向对象编程的限制去进行元信息的添加，使用得当可以大幅度提高框架的灵活性。

// CustomEventBus.java 只列出了修改的方法
public class CustomEventBus {
    public void post(Object event) {
        if(Objects.isNull(subscriberMap)
                || Objects.isNull(event)
                || Objects.isNull(subscriberMap.get(event.getClass())))
            return;
        // 获取该事件类型继承的父类和接口
        Set<Class<?>> postType = getSuperClassesAndInterfaces(event.getClass());
        postType.forEach((eventType) -> {
            Set<Subscriber> subscriberSet = subscriberMap.getOrDefault(eventType, Collections.emptySet());
            for(Subscriber subscriber : subscriberSet) {
                // 使用反射调用update方法
                subscriber.update(event);
            }
        });
    }

    // 获取所有父类以及接口
    private Set<Class<?>> getSuperClassesAndInterfaces(Class<?> eventType) {
        Set<Class<?>> visited = new HashSet<>();
        Queue<Class<?>> queue = new LinkedList<>();
        queue.add(String.class);
        while(!queue.isEmpty()) {
            Class<?> clazz = queue.poll();
            if(!visited.add(clazz)) continue;
            if(Objects.nonNull(clazz.getSuperclass()))
                queue.add(clazz.getSuperclass());
            queue.addAll(Arrays.asList(clazz.getInterfaces()));
        }
        return visited;
    }
}

实现继承事件路由的方法也是，在调用post()方法是获取事件类型的父类，从Map中查找全部父类并获取Subscriber，分别调用他们的处理方法。

上述代码将getSuperClassesAndIntefaces()方法作为私有方法放在了CustomEventBus中，但仔细思考以下，查找父类这一操作其实并不属于CustomEventBus的职责。所以，这一部分代码最好的处理方法是将之抽离，而Guava中就有用于获取某个Class全部父类和接口的反射用工具类。而我们在进行码业务逻辑代码时，当方法中的代码行数过多就要考虑抽取为一个方法了，当增加一个方法时就需要考虑这个方法是否应该属于这个类，这也就是封装的重要性。

多线程和策略模式

写到现在，上面的代码能够执行出看似合理的结果。但目前位置，我们所有的测试，包括subscribe()和post()的调用都是在单线程下进行的，如果放在多线程环境下，就必须考虑2个问题：

  1. 多线程调用subscribe()和post()方法时，如何保证线程安全
  2. 调用post()线程的方法和执行事件处理方法的线程一定要是同一个吗

subscribe()的调用安全其实就是管理Subscriber数据结构的访问安全，在上面的代码中，我们使用的是Map去进行管理，所使用的实现类是HashMap，这并非一个并发数据结构，所以肯定不行。

post()方法的多线程调用在上述的代码中会产生的问题是，调用post()的线程会执行全部的Subscriber的方法调用，而我们不能保证Subscriber的方法是线程安全的，所以就会产生并发的各种问题。

第2个问题在于我们不知道使用EventBus的人想要怎样的调用方式，因此我们就需要抽离出这一部分的变化。

class Subscriber {
    // 通过用户给出的方法的线程安全性创建不同的Subscriber实例
	static Subscriber create(EventBus bus, Object listener, Method method) {
        return (Subscriber)(isDeclaredThreadSafe(method) ? new Subscriber(bus, listener, method) : new Subscriber.SynchronizedSubscriber(bus, listener, method));
    }
    // 注册的方法是否通过@AllowConcurrentEvents注解声明为线程安全
    private static boolean isDeclaredThreadSafe(Method method) {
        return method.getAnnotation(AllowConcurrentEvents.class) != null;
    }
    @VisibleForTesting
    // 非线程安全的Subscriber
    static final class SynchronizedSubscriber extends Subscriber {
        private SynchronizedSubscriber(EventBus bus, Object target, Method method) {
            super(bus, target, method, null);
        }
        // 调用前给上个锁
        void invokeSubscriberMethod(Object event) throws InvocationTargetException {
            synchronized(this) {
                super.invokeSubscriberMethod(event);
            }
        }
    }
}

Guava在post()方法的线程安全方面，再次通过注解将这一元信息的描述交给了使用者。

abstract class Dispatcher {
    static Dispatcher perThreadDispatchQueue() {
        return new Dispatcher.PerThreadQueuedDispatcher();
    }
	
    static Dispatcher legacyAsync() {
        return new Dispatcher.LegacyAsyncDispatcher();
    }

    static Dispatcher immediate() {
        return Dispatcher.ImmediateDispatcher.INSTANCE;
    }
}

Dispatcher意思是分发器，它决定的是事件的分发顺序

首先来看看perThreadDispatchQueue，它的主要逻辑是通过一个ThreadLocal<Queue<...>>按照FIFO的顺序处理本线程的事件分发

其次是ImmediateDispatcher，这个分发器的逻辑是完成当前的事件处理方法再进行下一次处理，在大多数情况下，它的分发顺序和perThreadDispatchQueue是一样的，它们之间的区别在于事件处理方法中如果调用了EventBus.post()，这样形成的时间分发树是深度优先还是广度优先。详情查看stackoverflow

最后是LegacyAsyncDispatcher，这个分发器使用的是ConcurrentLinkedQueue<...>解决来自多线程的事件，因此分发顺序基于多线程竞争后在队列中的顺序。

Executor决定了事件分发后的执行线程究竟是谁，可以使用线程池异步去执行事件处理，也可以使用当前线程进行处理，不多赘述了。

final class SubscriberRegistry {
    // 缓存类中的@Subscribe标识的方法
	private static final LoadingCache<Class<?>, ImmutableList<Method>> subscriberMethodsCache = CacheBuilder.newBuilder().weakKeys().build(new CacheLoader<Class<?>, ImmutableList<Method>>() {
        public ImmutableList<Method> load(Class<?> concreteClass) throws Exception {
            return SubscriberRegistry.getAnnotatedMethodsNotCached(concreteClass);
        }
    });
    // 缓存类和其全部的父类和接口
    private static final LoadingCache<Class<?>, ImmutableSet<Class<?>>> flattenHierarchyCache = CacheBuilder.newBuilder().weakKeys().build(new CacheLoader<Class<?>, ImmutableSet<Class<?>>>() {
        public ImmutableSet<Class<?>> load(Class<?> concreteClass) {
            return ImmutableSet.copyOf(TypeToken.of(concreteClass).getTypes().rawTypes());
        }
    })
}

这两部分的LoadingCache用于减少反射带来的性能损耗，subscriberMethodsCache可以降低register()的参数为同一个类的对象时多次反射获取方法信息带来的性能损耗。而flattenHierarchyCache是在调用post()时，解析事件类型的继承结构而定义的缓存。

这两部分都是为了解决反射耗时的问题而存在的，至于反射有多耗时，大伙儿可以自己写一写测试一下。

总结

注解和反射作为Java区别于其他语言的两大特性，能够很好的帮助框架设计者做出更灵活的框架，但它们的使用远比上面所写的要复杂，并且反射也是一种比较“危险”的操作，在实际使用时需要慎重考虑并处理好相应的异常，否则很容易发生灾难。并且反射相比于常规手段来说更为耗时。

面向对象的设计方法很多都是和设计模式相挂钩，“策略模式”是一个十分典型且有效的设计模式，因而在许多框架和工具中都有体现。而面向对象的另一方面就是封装，我认为“单一职责”这一设计原则有时候很容易被忽略并且不好把握，所以写代码时经常就会出现不应该出现的“全能类”。

Guava作为类库而言是十分易用的，但内部的实现其实很多时候超乎预想的复杂，需要考虑的细节也比较多，我所写的也没有将之全部包含进去，感兴趣的小伙伴务必自己去看一看。Orz
