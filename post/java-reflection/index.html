Java - 反射

2020年7月3日
| Java
| 阅读

反射是Java的一大特点，能够在运行时获取类的全部信息，因此，在许多的框架中都有其应用。其中我们最熟悉的当属spring了，它的核心两大功能ioc和aop都有使用反射机制，今天我们就来聊一聊反射机制。

简单使用

反射是通过获取Class对象，进而在运行时获取类的全部成员、方法、注解、父类等信息，因而首先需要做的，就是获取Class对象。

上述的3种方法各有其用途，但我印象最深的是第三种，它拥有能够将字符串转换为类并创建实例的能力，这在C和C++中是难以想象的，只能通过手动实现类似工厂方法的途径做到。

        Class<?> clazz = HelloWorld.class;
        HelloWorld helloWorld = new HelloWorld();

        // 使用Method对象调用方法
        method.invoke(helloWorld);
    }

通过Class获取Method对象从而调用相应的方法，这也属于相对比较常见的操作。

反射的代价

异常代价

上面的2个例子中，所有的方法名后面都抛出了异常，这便是反射的代价之一。反射的api在赋予你运行时获取类相关信息能力的同时，也使得JVM放弃了它自己在编译时期对你代码的纠错能力，因此，你必须在代码中处理自己在运行时使用反射可能产生的异常。

这是因为你在运行时期所决定的东西，编译器在编译时期无法知晓。编译器不知道你会调用什么方法，也就不知道你调用方法的访问修饰符，更不知道调用方法的对象是否属于这个类。或许你会觉得“就这？”，为了处理这些异常，你就很容易写出下面这样的代码。

上述代码将所有的异常都进行了处理，但回过头来你就会发现，你处理异常所使用的代码行数超过了你的业务逻辑。这就导致了你所写的代码变得十分的杂乱，业务逻辑不清晰，也就影响了代码的可读性。

    @Test
    public void methodPerformanceTest() throws ClassNotFoundException, NoSuchMethodException, InvocationTargetException, IllegalAccessException {
        final long CIRCLES = 1000000;
        HelloWorld helloWorld = new HelloWorld();
        Stopwatch stopwatch = Stopwatch.createUnstarted();

        // 通过Class.forName()调用
        stopwatch.start();
        for(int i = 0; i < CIRCLES; ++i) {
            Class.forName("se.HelloWorld").getMethod("sayHello").invoke(helloWorld);
        }
        stopwatch.stop();
        System.out.println(stopwatch.elapsed(TimeUnit.MILLISECONDS));

        // 通过HelloWorld.class调用
        stopwatch.reset();
        stopwatch.start();
        for(int i = 0; i < CIRCLES; ++i) {
            HelloWorld.class.getMethod("sayHello").invoke(helloWorld);
        }
        stopwatch.stop();
        System.out.println(stopwatch.elapsed(TimeUnit.MILLISECONDS));

        // 直接调用
        stopwatch.reset();
        stopwatch.start();
        for(int i = 0; i < CIRCLES; ++i) {
            helloWorld.sayHello();
        }
        stopwatch.stop();
        System.out.println(stopwatch.elapsed(TimeUnit.MILLISECONDS));
    }
// ----- 输出 -----
680
137
4

上述代码通过3种方式调用sayHello()方法，最终在我的机器上得到的结果。Class.forName()的代价极高，在实际使用时反复调用会极大的影响程序的性能。而即使使用HelloWorld.class直接获取方法，与直接调用相比依然相差的30倍以上（实际上这个差距并不是固定的，可能换一个类或者换一个环境就会有所变化)。

对于需要反复调用的方法而言，这是无法接受的，因此，比较好的优化方案是通过保存Method对象，这样就不需要从Class对象中获取了。

    @Test
    public void methodPerformanceTest() throws ClassNotFoundException, NoSuchMethodException, InvocationTargetException, IllegalAccessException {
        final long CIRCLES = 1000000;
        HelloWorld helloWorld = new HelloWorld();
        Stopwatch stopwatch = Stopwatch.createUnstarted();

        // 从Class对象中获取Method对象
        stopwatch.start();
        for(int i = 0; i < CIRCLES; ++i) {
            HelloWorld.class.getMethod("sayHello").invoke(helloWorld);
        }
        stopwatch.stop();
        System.out.println(stopwatch.elapsed(TimeUnit.MILLISECONDS));

        // 保存Method对象
        Method helloMethod = Class.forName("se.HelloWorld").getMethod("sayHello");
        stopwatch.reset();
        stopwatch.start();
        for(int i = 0; i < CIRCLES; ++i) {
            helloMethod.invoke(helloWorld);
        }
        stopwatch.stop();
        System.out.println(stopwatch.elapsed(TimeUnit.MILLISECONDS));

        // 直接调用
        stopwatch.reset();
        stopwatch.start();
        for(int i = 0; i < CIRCLES; ++i) {
            helloWorld.sayHello();
        }
        stopwatch.stop();
        System.out.println(stopwatch.elapsed(TimeUnit.MILLISECONDS));
    }
// ----- 输出 -----
146
7
4

这次的结果我们相对满意，差距在2倍不到。但如果在多线程环境下，缓存Method对象需要付出的努力并不低

  • 需要考虑内存的容量问题
  • 需要考虑多线程访问的竞争问题

这就是Guava的BusEvent在使用注解和反射进行消息路由时所做的额外工作。因此，在业务逻辑中使用反射在大部分时候并不合适，一般其实可以思考更好的解决方法。

    @Test
    public void testPrivateMethod() throws NoSuchMethodException, InvocationTargetException, IllegalAccessException {
        HelloWorld helloWorld = new HelloWorld();
        // inaccessibleMethod是一个private方法
        Method inaccessibleMethod = HelloWorld.class.getDeclaredMethod("inaccessibleMethod");
        inaccessibleMethod.setAccessible(true);
        // 通过反射调用
        System.out.println(inaccessibleMethod.invoke(helloWorld));
    }

在通常情况下，用户是无法访问类的私有成员和方法的，但在反射下这一限制形同虚设。

private关键字的作用并非单纯的阻止用户访问私有成员。在我看来，public和private的区别在于前者是这个类所应该提供的功能，而后者是这些功能隐藏之下的细节。因此，对于这个类的开发者而言，private修饰的成员和方法相比与public而言能够去随意的删除和增加。因而对于这个类的使用者而言，当你使用反射访问private成员的同时，也增加了你自己代码的风险，或许类库升级一个版本，你的代码就不可用了。

还有一种情况是针对单例模式的类使用反射。大家都知道，单例模式通常都提供相应的工厂方法获取实例，但如果使用反射就会导致整个应用存在2个或多个实例，这也是由于反射破坏了类的封装性，访问了原本无法访问的构造方法。

反射的应用

接口和实现分离

比较常见的一个例子就是日志类slf4j，它定义了日志接口Logger，但没有定义任何实现。这原本是极为常见的，但我们获取日志实现的方式却并非创建Logger的实现类对象，并将之赋值给Logger引用，而是通过以下方式。

1

	private static final Logger log = LoggerFactory.getLogger(xxxController.class);

那么，问题就是由slf4j所定义的LoggerFactory是如何获取其他类库实现的Logger对象？答案便是通过反射

    static Set<URL> findPossibleStaticLoggerBinderPathSet() {
        LinkedHashSet staticLoggerBinderPathSet = new LinkedHashSet();

        try {
            ClassLoader loggerFactoryClassLoader = LoggerFactory.class.getClassLoader();
            Enumeration paths;
            if (loggerFactoryClassLoader == null) {
                // STATIC_LOGGER_BINDER_PATH = "org/slf4j/impl/StaticLoggerBinder.class";
                paths = ClassLoader.getSystemResources(STATIC_LOGGER_BINDER_PATH);
            } else {
                paths = loggerFactoryClassLoader.getResources(STATIC_LOGGER_BINDER_PATH);
            }

            while(paths.hasMoreElements()) {
                URL path = (URL)paths.nextElement();
                staticLoggerBinderPathSet.add(path);
            }
        } catch (IOException var4) {
            Util.report("Error getting resources from path", var4);
        }

        return staticLoggerBinderPathSet;
    }

反射能够通过JVM获取已经载入的类的信息，如果你在自己的代码中引入了Logger实现的以来，那么就能够被JVM找到。这也就是为什么能够统一的方法LoggerFactory.getLogger()获取不同日志实现，而我们自己的代码并不需要做任何修改。

与此类似的还有javax.validation仅定义了Validator的接口，而hibernate-validator实现了它，最终通过Validation.buildDefaultValidatorFactory().getValidator()能够获取实现类的实例。

反射的这种使用方式改变了工厂方法、接口、实现类之间的关系，使得前工厂方法的实现不需要依赖实现类来完成，最终的效果就是简化了我们的使用。

// 方法调用拦截
public class HelloInvocationHandler implements InvocationHandler {

    private Hello hello;

    public HelloInvocationHandler(Hello hello) {
        this.hello = hello;
    }

    public Object invoke(Object o, Method method, Object[] objects) throws Throwable {
        System.out.println("Before " + method.getName());
        Object result = method.invoke(hello, objects);
        System.out.println("After " + method.getName());
        return result;
    }
}
// 测试类
public class JdkProxyTest {
    @Test
    public void proxyTest() throws NoSuchMethodException {
        Hello helloProxy = (Hello) Proxy.newProxyInstance(
                Hello.class.getClassLoader(),
                new Class<?>[] {Hello.class},
                new HelloInvocationHandler(new HelloWorld())
        );
        // sayHello()打印"Hello world"
        helloProxy.sayHello();
    }
}

动态代理更上层的一个应用就是aop，通过定义切点、切面和通知类型就能够完成代理类的生成。而更上层的应用包括我们所熟知的spring声明式事务、Controller统一异常处理以及参数校验等功能。

动态代理是对OOP静态代理模式的一种拓展，它最大的优势在于无入侵的代码增强以及能够对某一类（接口或注解）对象进行代理。

注解

注解是一种Java给类、方法、成员、参数等提供元信息的语法。如果单纯的标注类，注解的行为类似于这个类继承了某一个接口，我觉得也是因为这个原因注解的定义才会是@interface（猜的）。

注解给我最大的印象是它能够灵活的修饰Java当中的各种级别的单元。这一点对比与接口也最为明显，接口作为OOP的代表特色，它只能修饰于类，而这造成的问题有

  • 就是接口是类级别的抽象，对方法抽象的唯一方法就是一个方法一个类
  • 限制了相应方法参数和返回值，这表示两者都必须进行抽象才能适应拓展

而注解能够直接作用与方法，通过反射直接获取被注解所标记的方法，实现方法级别的抽象。

近些年来注解大行其道，应用及其广泛，改变了很多原有的元信息赋予方式，比如spring的xml。除了spring之外，其他使用比较广泛的注解使用有：

总结

首先，反射作为使用Java解决问题的一个可选方案，不应该在任何需要解决问题的时候第一时间想到它，它所带来巨大灵活性的前提是需要考虑更多的细节以及付出相应的性能代价。因此，在能够使用其他不太复杂的方法解决问题的情况下应当避免使用反射。

其次，如果需要使用反射也需要避免直接使用。Java有着许多底层使用反射的上层方案，它们的存在能保证你在解决问题的同时不需要思考底层可能出现的问题，并且代码简洁和可读，把脏活累活交给别人，何乐而不为。
