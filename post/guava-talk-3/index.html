Guava杂谈 - LoadingCache

2020年7月1日
| Java
| 阅读

我与Cache这个词第一次相遇是在本科的《计算机组成原理》这门课上，这门课当时上起来那是满头问号。Cache是用于存储数据的一种组件，它存在的目的是让获取数据的速度更快。在集成Web服务时，使用比较广泛的Cache服务器应该是redis了，它作为跨进程甚至跨网络的缓存系统提供了十分丰富的功能。而今天谈论的Guava中的LoadingCache是仅在单个JVM实例中使用的缓存，而作为本地Cache，当存储数据量十分小并且没有持久化需求时可以在不增加整个应用复杂性的前提下进行使用。

Cache和ConcurrentMap

单纯从存储结构而言，Cache的一部分就是ConcurrentMap，这是由于后者提供了前者所需要的几个功能：

  • 快速查找。Cache的目的就是为了能够减少计算/读取所带来的时间开销，因而存储结构必须拥有快速查找的能力
  • 并发读写。Cache必须能够对并发有着很好的支持，不能单纯以加锁为手段将读写串行化，这样就没有意义

也因此在锁的设计上，Cache和ConcurrentHashMap一样使用了ReentrantLock和分段锁的设计增强了整体的并发能力。

但Cache所需要提供的功能却比ConcurrentMap要多，主要的差别是Cache中的内容会被淘汰。因此，为了满足使用，就必须提供比较灵活的淘汰策略，Guava所提供的淘汰策略有以下几种：

  • 时间淘汰策略
    • expireAfterAccess()。访问后经过一定时间将ReferenceEntry淘汰
    • expireAfterWrite()。写入后经过一定时间将ReferenceEntry淘汰
  • 容量淘汰策略
    • maximunSize()。设定Cache的最大容量，当容纳的数据将要超过最大容量时，通过LRU机制将之淘汰
  • 重量淘汰策略
    • maximumWeight()。设定Cache能够承受的最大重量，当要超过最大重量时，通过LRU淘汰数据
    • weigher()。给出计算数据重量的Weigher实现
  • 引用淘汰策略
    • weakKeys()。当没有key的强引用时，回收整个ReferenceEntry
    • weakValues()。当没有value的强引用时，回收整个ReferenceEntry
    • softValues()。当内存不足，且没有value的强引用时，回收整个ReferenceEntry

除此之外，Guava的Cache还拥有统计和刷新功能，用于观察Cache的缓存命中率等统计信息以及手动刷新数据。

            .build(new CacheLoader<Integer, String>() {
                @Override
                public String load(Integer key) throws Exception {
                    return System.currentTimeMillis() + " Value " + key;
                }
            });
    // manually put
    System.out.println("-------------- 手动载入数据 ------------");
    loadingCache.put(1, "Manual Value 1");
    System.out.println(String.format("The value of key %d is %s", 1, loadingCache.get(1)));

    System.out.println(String.format("The value of key %d is %s", 2, loadingCache.get(2)));

-------------- 自动载入数据 ------------

如上述代码输出所示，LoadingCache插入数据的方式有两种：：

  • 通过调用put()方法手动插入数据
  • 通过调用get()方法没有数据自动载入

@Test
public void testMaximumSize() throws Exception{
    LoadingCache<Integer, String> loadingCache = CacheBuilder.newBuilder()
            .maximumSize(2)
            .build(new CacheLoader<Integer, String>() {
                @Override
                public String load(Integer key) throws Exception {
                    return System.currentTimeMillis() + " Value " + key;
                }
            });
    // manually put
    loadingCache.put(1, "Manual Value 1");
    loadingCache.put(2, "Manual Value 2");
    loadingCache.get(1);
    loadingCache.put(3, "Manual Value 3");
    System.out.println(String.format("The value of key %d is %s", 2, loadingCache.getIfPresent(2)));
}
// ----- 输出 -----
The value of key 2 is null

上述代码的输出说明了：

  • 当插入的数据超过最大容量时条时，之前的数据会被淘汰
  • 淘汰策略为LRU，即由于1在最近被访问了，所以淘汰2

@Test
public void testExpireAfterAccess() throws Exception{
    LoadingCache<Integer, String> loadingCache = CacheBuilder.newBuilder()
            .expireAfterAccess(1, TimeUnit.SECONDS)
            .build(new CacheLoader<Integer, String>() {
                @Override
                public String load(Integer key) throws Exception {
                    return System.currentTimeMillis() + " Value " + key;
                }
            });
    // manually put
    loadingCache.put(1, "Manual Value 1");
    System.out.println("------- 1秒前 --------");
    System.out.println(String.format("The value of key %d is %s", 1, loadingCache.getIfPresent(1)));
    Thread.sleep(1000);
    System.out.println("------- 1秒后 --------");
    System.out.println(String.format("The value of key %d is %s", 1, loadingCache.getIfPresent(1)));
}
// ----- 输出 -----
------- 1秒前 --------
The value of key 1 is Manual Value 1
------- 1秒后 --------
The value of key 1 is null

如上述代码输出结果，在经过1s后，存入Cache的数据被淘汰了。

这里需要注意，Guava并没有通过使用额外的Cache维护线程去淘汰过期数据，而是在访问时检查该数据是否过期。这样做的原因是为了保证Cache的性能以及设计的简单性，不引入过度复杂的机制。

@Test
public void testValueReferenceEvict() throws Exception{
    LoadingCache<Integer, String> loadingCache = CacheBuilder.newBuilder()
            .weakValues()
            .build(new CacheLoader<Integer, String>() {
                @Override
                public String load(Integer key) throws Exception {
                    return System.currentTimeMillis() + " Value " + key;
                }
            });
    // manually put
    loadingCache.put(128, new String("Manual Value 128"));
    System.out.println("------- GC前 --------");
    System.out.println(String.format("The value of key %d is %s", 128, loadingCache.getIfPresent(128)));
    System.gc();
    System.out.println("------- GC后 --------");
    System.out.println(String.format("The value of key %d is %s", 128, loadingCache.getIfPresent(128)));
}
// ----- 输出 -----
------- GC前 --------
The value of key 128 is Manual Value 128
------- GC后 --------
The value of key 128 is null

上述代码是使用weakValues()将值引用设置为弱引用，因此在GC前能够正常获取，而在GC后值会被回收，而整个ReferenceEntry会被Cache回收。

上面的代码使用new String()而不是直接使用字符串的原因是为了防止字符串常量池一直持有引用对结果的影响。同样，如果使用基本类型的包装类，需要防止包装类缓存了一定范围的数值，比如Integer缓存了-128至127的Integer对象，这样的对象是一直被强引用的。

@Test
public void testKeyReferenceEvict() throws Exception{
    LoadingCache<Integer, String> loadingCache = CacheBuilder.newBuilder()
            .weakKeys()
            .build(new CacheLoader<Integer, String>() {
                @Override
                public String load(Integer key) throws Exception {
                    return System.currentTimeMillis() + " Value " + key;
                }
            });
    // manually put
    Integer key = 128;
    loadingCache.put(key, "Manual Value 128");
    System.out.println("------- 持有引用 --------");
    System.out.println(String.format("The value of key %d is %s", key, loadingCache.getIfPresent(key)));
    key = 128;
    System.out.println("------- 不持有引用 --------");
    System.out.println(String.format("The value of key %d is %s", key, loadingCache.getIfPresent(key)));
}
// ----- 输出 -----
------- 持有引用 --------
The value of key 128 is Manual Value 128
------- 不持有引用 --------
The value of key 128 is null

这里的代码输出结果比较有意思，因为我没有调用System.gc()进行垃圾回收key，但得到的结果依然是null。这是由于Guava的Cache对非强引用的key和value使用的是==比较，而不是equals()。我们获取value通过传递key进行查找，因此单单使用weakValues()我们不会感觉到这点不同，但当我们使用weakKeys()时，由于使用了key = 128替换了对象导致引用不同，所以得到的结果为null。

因此，如果使用了weakKeys()，切记使用原本的对象进行查询操作

引用淘汰设计

上述提及的Cache功能当中，最为有意思的就是通过引用强度的使用去回收缓存数据了，首先就存在一下几个问题：

  • 如何根据key和value的引用强度不同去创建不同的ReferenceEntry
  • 回收key或者value的是GC，那么如何回收整个ReferenceEntry
  • 如何和其他缓存淘汰机制产生影响

enum Strength {
	// 强引用
	STRONG {
	    @Override
	    <K, V> ValueReference<K, V> referenceValue(...) {...}
	    @Override
	    Equivalence<Object> defaultEquivalence() {...}
	},
	// 软引用
	SOFT {
	    @Override
	    <K, V> ValueReference<K, V> referenceValue(...) {...}
	    @Override
	    Equivalence<Object> defaultEquivalence() {...}
	},
	// 弱引用
	WEAK {
	    @Override
	    <K, V> ValueReference<K, V> referenceValue(...) {...}
	    @Override
	    Equivalence<Object> defaultEquivalence() {...}
	};

	// 通过强度决定value所使用的引用类型
	abstract <K, V> ValueReference<K, V> referenceValue(
		Segment<K, V> segment, ReferenceEntry<K, V> entry, V value, int weight);

	// 强度所决定的对比器
	abstract Equivalence<Object> defaultEquivalence();
}

上面的枚举提供了两个抽象方法，根据强度有各自的实现

  • valueReference()能够根据强度决定value所使用的引用类型
  • defaultEquivalence()获取强度对应的对比器

因此，这部分的美剧仅仅只能决定value的引用使用了ValueReference，并且根据强度所使用的具体实现不同（上面的代码省略了），一共有6种：

  • StrongValueReference和WeightedStrongValueReference强引用下的2种value引用。
  • SoftValueReference和WeightedSoftValueReference软引用下的2种value引用。继承SoftReference
  • WeakValueReference和WeightedWeakValueReference弱引用下的2种value引用。继承WeakRefercent

value的引用类型受到Cache是否设置了最大重量淘汰的影响，当设置了该淘汰策略后，会在WeightedxxxValueReference中记录相应的重量信息。

static final EntryFactory[] factories = {
    STRONG,
    STRONG_ACCESS,
    STRONG_WRITE,
    STRONG_ACCESS_WRITE,
    WEAK,
    WEAK_ACCESS,
    WEAK_WRITE,
    WEAK_ACCESS_WRITE,

// 通过key的强度和后面两个属性决定使用的EntryFactory
static EntryFactory getFactory(
	Strength keyStrength, boolean usesAccessQueue, boolean usesWriteQueue) {
	int flags =
		((keyStrength == Strength.WEAK) ? WEAK_MASK : 0)
		    | (usesAccessQueue ? ACCESS_MASK : 0)
		    | (usesWriteQueue ? WRITE_MASK : 0);
	return factories[flags];
}

上面代码的逻辑十分清晰，就是通过key强度、usesAccessQueue以及usesWriteQueue3个属性从表中查出对应的EntryFactory对象。问题在于我们明白key强度所表示的信息，却不明白后面2个属性所代表的含义，这也包含了同一引用强度下工厂对象产生的EntryReference有什么区别，因此看一看传递过来的相关源码：

    return maxWeight >= 0;

boolean usersAccessEntries() {
    return (expireAfterAccessNanos > 0 || maxWeight >= 0) || expireAfterAccessNanos > 0;
}

这就是usesAccessQueue的决策过程，可以很明显的看出，影响所使用的EntryFactory的是是否设置了访问时间淘汰（expireAfterAccess()）以及重量/容量淘汰策略（maximumSize()或maximumWeight()），这是由于这两种淘汰策略都涉及了LRU，因此访问队列会决定哪个缓存数据会被淘汰。

相应的usesWriteQueue属性取决于是否设置了写入时间淘汰（expireAfterWrite()）以及写入后自动刷新（refreshAfterWrite()）。

  // WeakEntry继承自WeakReference，表示key的强度
  static class WeakEntry<K, V> extends WeakReference<K> implements ReferenceEntry<K, V> {
  }

  static final class WeakAccessWriteEntry<K, V> extends WeakEntry<K, V> {
    // 访问时间，volatile保证多线程的读取一致
    volatile long accessTime = Long.MAX_VALUE;
    // 访问时间队列双向链表，下一个Entrys
    ReferenceEntry<K, V> nextAccess = nullEntry();
    // 访问时间队列双向链表，上一个Entry
    ReferenceEntry<K, V> previousAccess = nullEntry();

    // 写入时间，volatile保证多线程的读取一致
    volatile long writeTime = Long.MAX_VALUE;
    // 写入时间队列双向链表，下一个Entrys
    ReferenceEntry<K, V> nextWrite = nullEntry();
    // 写入时间队列双向链表，上一个Entry
    ReferenceEntry<K, V> previousWrite = nullEntry();
  }

上述代码就表明了WeakAccessWriteEntry对应的结构

  • key的强度决定了ReferenceEntry的强度
  • 根据是否需要访问队列决定ReferenceEntry是否是双向链表

综上，我们基本解决了第一个和第三个问题：

  • 如何根据key和value的引用强度不同去创建不同的ReferenceEntry
    • key和value的引用强度分别决定了ReferenceEntry的实现以及ValueReference的实现，而这两者通过继承不同的引用而产生了不同的回收效果
  • 如何和其他缓存淘汰机制产生影响
    • 所有的淘汰机制都是通过EntryReference和ValueReference决定的，不同的淘汰机制会使用不同的实现进行组合

V get(K key, int hash, CacheLoader<? super K, V> loader) throws ExecutionException {
	checkNotNull(key);
	checkNotNull(loader);
	try {
		// 检查Segment中存在的EntryReference数量
		if (count != 0) { // read-volatile
			// 获取ReferenceEntry
			ReferenceEntry<K, V> e = getEntry(key, hash);
			if (e != null) {
				long now = map.ticker.read();
				// 获取“活的”Value
				V value = getLiveValue(e, now);
				if (value != null) {
					recordRead(e, now);
					statsCounter.recordHits(1);
					return scheduleRefresh(e, key, hash, value, now, loader);
				}
				ValueReference<K, V> valueReference = e.getValueReference();
				if (valueReference.isLoading()) {
					return waitForLoadingValue(e, key, valueReference);
				}
			}
		}
	}
}

@GuardedBy("this")
// 清理keyReferenceQueue
void drainKeyReferenceQueue() {
	Reference<? extends K> ref;
	int i = 0;
	while ((ref = keyReferenceQueue.poll()) != null) {
		@SuppressWarnings("unchecked")
			ReferenceEntry<K, V> entry = (ReferenceEntry<K, V>) ref;
		map.reclaimKey(entry);
		if (++i == DRAIN_MAX) {
			break;
		}
	}
}

@GuardedBy("this")
// 清理keyReferenceQueue
void drainValueReferenceQueue() {
	Reference<? extends V> ref;
	int i = 0;
	while ((ref = valueReferenceQueue.poll()) != null) {
		@SuppressWarnings("unchecked")
			ValueReference<K, V> valueReference = (ValueReference<K, V>) ref;
		map.reclaimValue(valueReference);
		if (++i == DRAIN_MAX) {
			break;
		}
	}
}

这两部分的代码逻辑十分简单，问题在于它是什么时候将无效的EntryReference或者ValueReference加入相应的队列的呢，最终通过查看构造方法，发现这两个队列是交给WeakReference进行使用的。因此，它们是在WeakReference在其引用的内容被GC回收后自动追加到队列中的。

由此，我们就解决了第2个问题。Guava通过向WeakReference或者SoftReference注册ReferenceQueue实例，当发生GC导致对象被回收后，通过队列后去相应的ReferenceEntry，最后将之从Cache中移除。

总结

Cache的实现灵活使用了Java的enum和Reference相关的内容。Java的enum其实相当灵活，不仅仅能够拥有各自的数据，还能够拥有各自的实现。enum中的对象就如同继承了同一个接口的类的单例，实际开发中可以结合EnumSet和EnumMap进行各种灵活使用。其次是Java中的引用，软、弱、虚3种引用我在以前的使用中并没有太多的接触，不知道能够通过ReferenceQueue去接收被回收的引用以及相应的实际应用场景，今天算补上了一点。

Guava的Cache实现提供了相当丰富的功能，其源码内容除了上述之外还涉及很多多线程的内容，感兴趣的可以自行学习。
